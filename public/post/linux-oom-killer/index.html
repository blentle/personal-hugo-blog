 1    /proc/[pid]/oom_score_adj (since Linux 2.6.36)
 2    This  file can be used to adjust the badness heuristic used to select which process
 3    gets killed in out-of-memory conditions.
 4
 5    The badness heuristic assigns a value to each candidate task ranging from 0  (never
 6    kill)  to 1000 (always kill) to determine which process is targeted.  The units are
 7    roughly a proportion along that range of allowed memory the  process  may  allocate
 8    from, based on an estimation of its current memory and swap use.  For example, if a
 9    task is using all allowed memory, its badness score will be 1000.  If it  is  using
10    half of its allowed memory, its score will be 500.
11
12    There  is  an  additional  factor included in the badness score: root processes are
13    given 3% extra memory over other tasks.
14
15    The amount of "allowed" memory depends on the context in which the  OOM-killer  was
16    called.   If it is due to the memory assigned to the allocating task's cpuset being
17    exhausted, the allowed memory represents the set of mems assigned  to  that  cpuset
18    (see  cpuset(7)).   If  it  is  due  to  a mempolicy's node(s) being exhausted, the
19    allowed memory represents the set of mempolicy nodes.  If it is  due  to  a  memory
20    limit  (or  swap limit) being reached, the allowed memory is that configured limit.
21    Finally, if it is due to the entire system being out of memory, the allowed  memory
22    represents all allocatable resources.
23
24    The  value  of  oom_score_adj  is  added  to the badness score before it is used to
25    determine   which   task   to   kill.    Acceptable   values   range   from   -1000
26    (OOM_SCORE_ADJ_MIN)  to  +1000 (OOM_SCORE_ADJ_MAX).  This allows user space to conâ€
27    trol the preference for OOM-killing, ranging from always preferring a certain  task
28    or  completely disabling it from OOM-killing.  The lowest possible value, -1000, is
29    equivalent to disabling OOM-killing entirely for that task, since  it  will  always
30    report a badness score of 0.
31
32    Consequently,  it  is  very simple for user space to define the amount of memory to
33    consider for each task.  Setting a oom_score_adj value of  +500,  for  example,  is
34    roughly  equivalent  to  allowing  the  remainder of tasks sharing the same system,
35    cpuset, mempolicy, or memory controller resources to use at least 50% more  memory.
36    A  value of -500, on the other hand, would be roughly equivalent to discounting 50%
37    of the task's allowed memory from being considered as scoring against the task.
38
39    For backward compatibility with previous kernels, /proc/[pid]/oom_adj can still  be
40    used to tune the badness score.  Its value is scaled linearly with oom_score_adj.
41
42    Writing  to  /proc/[pid]/oom_score_adj or /proc/[pid]/oom_adj will change the other
43    with its scaled value.
